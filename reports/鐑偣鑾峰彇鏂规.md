# 热点详情采集落地实施方案

**日期**：2026/06/14
**性质**：核心平台热点详情采集方案

---

## 1. 调研结果与背景

### 1.1 现有技术方案分析

目前市面上的热榜采集工具可分为三类：

| 方案类型 | 代表工具/实现 | 能力范围 | 局限性 |
|---------|--------------|---------|---------|
| **公开热榜 API** | DailyHotApi、各平台官方热榜接口 | 获取热点标题、摘要、热度值 | 只能做召回，发现≠详情 |
| **通用网页抓取** | Jina Reader、Web scraping API | 抓取公开页面的标题/正文/链接等表层信息 | 遇到登录拦截直接失效，无法获取微博/小红书详情 |
| **浏览器 Agent** | Browser-use、Playwright + LLM Agent | 模拟用户操作，可处理动态页面 | 需要稳定 session，登录失效仍需人工介入 |

### 1.2 为什么舍弃通用方案

**Jina 等抓取工具**：对微博、小红书无效

- 微博搜索页未登录直接返回验证码页，抓取到的是验证拦截 HTML，不是正文
- 小红书详情页未登录返回 403，Jina 只能读到错误响应
- 即使加上 JavaScript 渲染能力，微博的滑块验证码、短信二次验证无法自动化绕过

**Browser Agent 方案**：session 维护仍是核心瓶颈

- Agent 可以模拟登录操作，但遇到滑块验证码同样无法通过
- session 过期后仍需要人工重新登录，自动化程度受限于登录态
- 维护成本高，收益有限，不适合第一版实现

### 1.3 本方案定位

**结论**：在现有技术条件下，微博和小红书的详情采集无法做到完全自动化。必须人工介入登录环节。

本方案采用**混合架构**：

- 基础热榜召回：DailyHotApi（HTTP 全自动）
- 详情采集：按平台能力分三种处理方式（见平台具体方案）
- 内容整理：LLM（用于生成简报，非必须，第一版用确定性 fallback）
- 输出：JSON + Markdown 报告

## 2. 整体逻辑与 Pipeline

### 2.1 整体 Pipeline 架构

```
DailyHotApi（召回）
    ↓
话题聚类（按标题去重合并）
    ↓
分平台详情采集（核心来源）
    ↓
Jina / 辅助抓取（辅助来源）
    ↓
LLM 整理（生成简报）
    ↓
输出 JSON + Markdown 报告
```

### 2.2 各环节说明

**(a) 使用 DailyHotApi 做热榜召回**

- 覆盖多平台（微博、百度、知乎、36氪、GitHub 趋势等）
- HTTP 接口，无需登录，响应快
- 负责产出 `HotRecord`：标题、摘要、热度值、URL
- **注意**：DailyHotApi 只做召回层，不做详情层。单独依赖它的话题永远不完整。

**(b) 调用 Jina 做辅助抓取**

- 用于辅助平台（GitHub 趋势、IT之家、36氪等）的正文抓取
- 这些平台无需登录，Jina 可直接拿到可读 HTML
- 作为辅助证据补充上下文，不参与核心完整性评估

**(c) 使用 LLM 进行处理**

- **第一版（确定性 fallback）**：直接拼接 evidence 内容生成摘要，不调用 LLM
- **长期版**：接入 LLM client，对 evidence 进行摘要、关键事实提取、话题组织
- LLM 作为可选扩展，不影响核心 pipeline 运行

**(d) 最后生成 JSON 数据**

Pipeline 输出四类文件：

| 文件 | 内容 |
|------|------|
| `data/raw/dailyhot_records.json` | 原始热榜记录 |
| `data/evidence/detail_evidence.json` | 所有平台的详情证据 |
| `data/processed/topic_briefs.json` | 整理后的话题简报 |
| `reports/core_platform_topic_digest.md` | 最终 Markdown 报告 |

## 3. 平台具体方案设计

### 3.1 各平台采集方式总览

| 平台 | 采集方式 | 自动化程度 | 人工介入环节 |
|------|---------|-----------|-------------|
| 百度搜索 | 直接 HTTP 请求 | **全自动** | 无 |
| 微博 | Playwright + 已登录浏览器会话 | **半自动** | 首次登录、session 失效恢复 |
| 小红书 | Playwright + 已登录浏览器会话 | **半自动** | 首次登录、session 失效恢复 |

### 3.2 百度搜索：全自动方案

**采集流程**：

```
Python HTTP 请求
    ↓
构建百度搜索 URL: https://www.baidu.com/s?wd={query}
    ↓
User-Agent 伪装
    ↓
请求 → 解析 HTML 中的搜索结果（title、snippet、url）
    ↓
存入 DetailEvidence
```

**实现要点**：

- 使用 `urllib.request` 或 `requests`，每次请求间隔 2~3 秒
- 若返回验证码页面（HTML 含"验证"字样），标记 `captcha_required`，跳过
- 解析使用正则或 BeautifulSoup，提取搜索结果条目

### 3.3 微博：半自动方案（人工介入为核心）

**关键限制**：微博搜索页未登录直接返回验证码，无任何公开接口可绕过。

**采集流程**：

```
第一步：人工登录（仅首次，之后定期维护）
    ↓
第二步：自动化采集（session 有效期内可重复执行）
    ↓
第三步：session 失效检测与人工恢复
```

**第一步：人工登录（一次性）**：

1. 运行 `make login-weibo`，Playwright 打开浏览器
2. 若无 session 文件，浏览器打开微博登录页
3. **人工输入账号密码，完成滑块验证码/二次验证**
4. Playwright 将 cookies 保存至 `data/browser_state/weibo.json`

**第二步：自动化采集（session 有效期内）**：

```
读取 session → 加载浏览器 profile
    ↓
打开微博搜索页: https://s.weibo.com/weibo?q={话题关键词}
    ↓
等待页面加载
    ↓
检测是否出现登录拦截
    ├─ 是 → 标记 login_required，保存截图
    └─ 否 → 提取前 5 条微博正文
    ↓
保存 HTML 快照 + 截图
    ↓
关闭浏览器
```

**第三步：session 失效人工恢复**：

```bash
make login-weibo
# 人工重新登录
make check-sessions
make run-core-pipeline
```

**session 有效期**：预计 7~30 天。

### 3.4 小红书：半自动方案（与微博一致）

**关键限制**：小红书详情页未登录返回 403，无公开接口。

**流程与微博完全一致**，仅 session 文件名不同（`xiaohongshu.json`）。

**session 失效恢复**：

```bash
make login-xiaohongshu
make check-sessions
make run-core-pipeline
```

## 4. 附录：详细机制说明

### 4.1 完整 Pipeline 触发顺序

```bash
make check-sessions          # 检查 session 状态
make login-weibo             # session 失效则重新登录
make login-xiaohongshu

make run-core-pipeline       # 执行完整 pipeline
```

**Pipeline 内部执行顺序**：

1. `collect-dailyhot` — 拉取热榜（全自动）
2. `collect-core-details` — 采集微博/小红书/百度详情
3. `collect-aux-evidence` — 补充辅助证据（全自动）
4. `cluster-topics` — 话题聚类（全自动）
5. `generate-briefs` — 生成简报（全自动）
6. `render-report` — 输出 Markdown（全自动）

人工介入只在 session 失效时发生，其余环节全自动。

### 4.2 人工介入环节汇总

| 环节 | 介入方式 | 频率 |
|------|---------|------|
| 微博首次登录 | 人工输入账号密码，触发验证码 | 仅首次 |
| 小红书首次登录 | 人工输入账号密码，触发验证码 | 仅首次 |
| 微博 session 失效恢复 | 重新运行 `make login-weibo` 并人工登录 | 预计每 7~30 天一次 |
| 小红书 session 失效恢复 | 重新运行 `make login-xiaohongshu` 并人工登录 | 预计每 7~30 天一次 |
| session 过期预警 | 人工监控 `make check-sessions` 输出 | 每周检查一次 |
| 采集结果抽检 | 人工打开截图检查内容是否有效 | 每次 pipeline 运行后 |

**最关键的人工介入点：微博和小红书的首次登录及 session 失效后的重新登录。**

### 4.3 截图与快照管理

每次采集自动保存（均不进入 Git）：

```
data/snapshots/weibo/{evidence_id}.html
data/screenshots/weibo/{evidence_id}.png
```

用途：人工抽检时直接打开截图确认内容是否正确。

### 4.4 异常处理规则

| 异常场景 | 处理方式 |
|---------|---------|
| 微博/小红书 session 失效 | 标记 `login_required`，pipeline 继续，记录到报告 |
| 百度触发验证码 | 标记 `captcha_required`，跳过该话题的百度证据 |
| 页面加载超时（>30s） | 标记 `failed`，保存截图供人工排查 |
| 网络错误 | 标记 `failed`，error_type 记录具体错误 |
| 采集到空内容 | 标记 `empty_content`，不影响其他平台 |
