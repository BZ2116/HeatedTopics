# 热点话题采集与标签化项目阶段报告

## 1. 项目背景与最终目标

本项目的最终目标，是构建一个可以持续自动化运行的热点话题采集、详情整理与标签生成工具。它后续可以封装成 Skill 或 Agent，每天或每周自动采集热点话题及其详细信息，整理成结构化数据，并为另一个创作者项目提供准确、可追溯、可检索的话题标签。

从业务角度看，这个项目不是单纯做“热榜搬运”，而是要解决创作者在选题时的三个问题：

1. 当前哪些话题正在变热。
2. 这些话题具体在讲什么，而不只是一个标题。
3. 这些话题应该被打上什么标签，适合哪些创作者、内容形式和受众。

因此，系统需要完成从“热点发现”到“详情证据”再到“标签索引”的完整链路。

## 2. 核心用途

当前项目的核心用途是：通过自动化工具持续获取热点，并为创作者提供精准的话题引导和标签参考。

具体来说，系统产出的结果会服务于下游创作者项目，作为创作者创作内容时的正确话题标签来源。下游项目可以基于这些标签完成：

- 热点选题推荐
- 创作者垂类匹配
- 内容方向筛选
- 话题标签建议
- 热点趋势跟踪
- 高风险话题提醒

为了让这些标签真正可用，项目不能只依赖热榜标题。标题只能说明“有这个热点”，但不能说明“这个热点到底是什么、为什么热、适合谁创作、创作时有什么风险”。所以当前项目重点建设的是热点详情采集和证据化整理能力。

## 3. 前期调研：市面上的热点 API 与 Agent 能力

### 3.1 国内热点 API

国内的热点数据源主要以“热榜聚合 API”为主，典型代表包括 DailyHotApi、今日热榜 TopHub/榜眼数据、ALAPI 今日热榜，以及部分云厂商或平台提供的热点播报接口。

这类工具的共同特点是：

- 覆盖中文互联网平台较多。
- 能快速获得平台、标题、排名、热度、链接等信息。
- 适合做第一层热点候选池。
- 通常无法直接提供完整事件详情。
- 对小红书、微博等社交平台的正文级数据支持有限。

代表性调研结果如下：

| 工具/API | 特点 | 适合用途 | 局限 |
| --- | --- | --- | --- |
| DailyHotApi | 开源热榜聚合 API，支持本地部署、RSS 和多平台路由 | 第一层热点发现、快速原型、自主可控采集 | 主要提供标题、热度、链接等，详情不足 |
| TopHub/榜眼数据 | 商业化热榜数据开放平台，提供榜单、榜单详情、历史快照 | 稳定商业接口、历史趋势分析 | 商业化依赖较强，仍偏热榜层 |
| ALAPI 今日热榜 | 聚合各站热点，接口文档和在线测试较完整 | 快速接入热榜数据 | 依赖 token、额度和商业服务稳定性 |
| 阿里云热榜播报类接口 | 偏热榜摘要和播报内容 | 可作为补充热点摘要来源 | 不一定覆盖我们需要的社交平台详情 |

结论：国内 API 普遍适合“发现热点”，但不适合单独承担“理解热点”和“生成创作者标签”的任务。

### 3.2 国外趋势与搜索 API

国外更多是趋势搜索、新闻搜索和 Agent Search API，而不是中文平台热榜聚合器。

代表包括：

| 工具/API | 特点 | 适合用途 | 局限 |
| --- | --- | --- | --- |
| Google Trends | 提供搜索趋势和地区热度 | 海外搜索趋势观察 | 官方更偏产品形态，后端 API 能力有限 |
| GDELT | 全球新闻、事件和媒体数据库，提供实时 JSON API | 国际新闻、严肃事件、多源媒体分析 | 对中文社交平台热榜覆盖不足 |
| Tavily | 面向 AI Agent 的搜索 API | Agent 检索、网页搜索、RAG 补充 | 不是热榜聚合器 |
| Exa | 面向语义搜索和网页内容获取 | 深度网页搜索、相似内容发现 | 中文平台热榜能力不是核心 |
| Perplexity/Sonar 类搜索 API | 面向问答和实时搜索 | 事实补充、摘要和核验 | 结果可解释性和成本需要评估 |

结论：国外工具更适合作为后续的“详情补充”和“事实核验”能力，而不是当前中文热点采集的第一入口。

### 3.3 Skill / Agent 生态调研结论

现有 Agent 或搜索类工具通常可以完成“给定问题后搜索网页并总结”，但它们大多不直接解决我们的问题。

我们的任务有几个特殊点：

1. 需要持续运行，而不是单次问答。
2. 需要每天或每周自动采集，而不是人工输入一个话题。
3. 需要面向中文平台热榜，尤其是微博、百度、小红书等。
4. 需要保留原始证据，不能只生成一段总结。
5. 需要输出稳定标签，供另一个项目程序化消费。

因此，通用搜索 Agent 可以作为后续增强模块，但当前更适合先建设自己的稳定数据管道，再逐步封装成专用 Skill 或 Agent。

## 4. 为什么当前选择 DailyHot

当前项目选择 DailyHotApi 作为第一层热点发现入口，主要原因是它适合当前阶段快速、可控地搭建热点候选池。

DailyHotApi 的优势包括：

- 开源，可本地部署，调试和扩展成本低。
- 覆盖平台较多，能统一获取多个平台的热榜。
- 返回结构相对统一，包含标题、排名、热度、链接、摘要、封面等字段。
- 接入简单，适合快速跑通从采集到报告的闭环。
- 不强依赖商业接口额度，适合项目早期迭代。

但我们并不把 DailyHot 当作唯一事实来源。它在项目中的定位是“热点发现”，不是“事实确认”。

原因是：

- 热榜 API 通常只提供话题名称或短摘要。
- 热榜标题容易有标题党、缩写、情绪化表达。
- 同一事件在不同平台可能有多个标题。
- 榜单数据无法直接说明事件背景、争议点、时间线和真实来源。
- 小红书这类平台经常只显示话题标题，缺少笔记正文和讨论上下文。

所以当前采用的是分层策略：

1. DailyHot 负责发现候选热点。
2. 百度、微博、小红书、Bilibili、掘金等平台负责补充详情。
3. 后续再引入新闻源、搜索 API 或 Agent Search API 做事实核验和多源交叉验证。

TopHub/榜眼数据、ALAPI 等工具后续可以作为补充或替代方案。当前阶段优先使用 DailyHot，是因为它开源、可控、成本低，更适合先把核心流程跑通。

## 5. 当前采取的方法

当前项目已经形成了一个五段式流程：

```text
多平台热榜发现
→ 话题去重与聚类
→ 详情信息采集
→ 证据结构化保存
→ 标签分类与报告输出
```

### 5.1 热点发现

热点发现阶段主要使用 DailyHotApi 获取多平台热榜数据。

当前覆盖的平台分为几类：

- 核心热点平台：微博、百度、知乎、今日头条
- 新闻补充平台：新浪新闻、澎湃、腾讯新闻、网易新闻
- 科技商业平台：36Kr、IT之家、掘金、CSDN
- 技术社区平台：GitHub、HelloGitHub
- 内容热度平台：Bilibili、抖音、快手
- 小红书补充来源：今日热榜小红书页、TopHub 小红书热榜

采集结果保存为 `HotRecord`，包含平台、标题、排名、热度、链接、封面、摘要、采集时间和原始载荷。

### 5.2 话题去重与聚类

热榜数据中经常出现重复话题。例如同一个事件可能同时出现在微博、百度、头条，也可能标题略有差异。

当前项目会对标题做基础规范化处理，例如去除空白、统一大小写、去掉部分热搜装饰词，再形成话题聚类。

聚类结果会保存：

- 统一话题标题
- 相关热榜记录 ID
- 相关证据 ID
- 覆盖平台
- 核心平台详情完整度
- 聚类置信度

这一步的目的是避免下游标签系统把同一事件当成多个不同话题。

### 5.3 详情信息采集

这是当前项目最关键的部分。

现有工具大多只能获取热点话题名称，无法直接获取详情。比如小红书榜单常常只有一个标题，如果不进入相关笔记，就无法知道用户到底在讨论什么。创作者项目如果只拿标题做标签，会出现标签不准确、话题误判、内容方向偏差等问题。

因此，当前项目为每个热点话题补充详情证据。

不同平台采用不同策略：

| 平台 | 详情采集方式 | 说明 |
| --- | --- | --- |
| 百度 | 基于标题生成搜索词，解析搜索结果，并尽量抓取可读网页正文 | 用于补充事件背景和外部来源 |
| 微博 | 使用 Playwright 复用本地登录态，访问搜索页并抽取帖子、页面文本和 DOM 内容 | 用于获取社交讨论现场 |
| 小红书 | 当前采集话题、热度、排名，并写入详情占位 | 后续由外部笔记采集项目补齐正文 |
| Bilibili | 使用热榜元数据补充视频标题、简介、链接 | 作为视频内容热度信号 |
| 掘金 | 使用 DailyHot 元数据补充技术类详情 | 作为技术话题补充证据 |

百度详情采集会生成多组查询词，例如：

```text
原始标题
原始标题 怎么回事
原始标题 最新进展
```

这样做的原因是，单独搜索标题可能只能返回热榜页，而加上“怎么回事”“最新进展”更容易召回事件解释、新闻背景和时间线信息。

### 5.4 证据结构化保存

项目不会只把详情写进最终报告，而是把每条详情保存为独立证据。

每条详情证据使用 `DetailEvidence` 结构，包含：

- `evidence_id`：证据 ID
- `topic_key`：对应话题
- `platform`：来源平台
- `source_method`：采集方式
- `query`：使用的查询词
- `url`：来源链接
- `title`：详情标题
- `content`：正文或详情内容
- `metrics`：热度、排名、搜索结果数量等指标
- `result_urls`：相关结果链接
- `fetched_at`：采集时间
- `fetch_status`：采集状态
- `error_type`：失败原因
- `confidence`：置信度
- `raw_payload`：原始载荷

这样做的好处是：

- 后续可以回溯信息来源。
- 可以区分成功、失败、空内容、低置信度。
- 可以对不同来源进行交叉验证。
- 报告和标签不是凭空生成，而是有证据链支撑。

### 5.5 语义标签引擎与创作者索引

标签分类不是简单地给热点标题贴几个关键词，而是项目中的“语义理解与创作者意图对齐层”。它的作用是把杂乱的热榜内容，转化成下游创作者项目可以稳定消费的话题资产。

当前设计可以理解为三层：

```text
话题语义理解层
→ 标签治理层
→ 创作者意图匹配层
```

第一层是话题语义理解。系统会综合热榜标题、平台元数据、详情正文、搜索结果、帖子内容和证据链接，判断这个热点到底属于什么事件、包含哪些实体、发生了什么动作、是否存在争议或风险。这里关注的是“理解话题”，而不是只匹配标题。

第二层是标签治理。系统把标签分成两类：一类是受控标签，一类是动态关键词。受控标签用于稳定分类，不能随意生成；动态关键词用于补充召回，可以随热点内容变化。

受控标签包括：

- `domain_path`：垂类路径，例如“教育升学 > 高考 > 分数线”
- `content_modes`：内容生产方式，例如“数据整理”“经验攻略”“实时跟进”“案例复盘”
- `audience_tags`：目标受众，例如“学生”“家长”“技术从业者”“泛大众”
- `risk_level`：话题风险等级
- `traceability`：证据可追溯程度

动态关键词包括：

- `entity_keywords`：人物、地点、品牌、产品、机构、考试名称等实体
- `event_keywords`：事件动作，例如“分数线公布”“政策发布”“案件通报”
- `match_terms`：面向检索召回的组合短语

第三层是创作者意图匹配。系统会把话题和创作者可能的内容需求建立联系，不只是判断“这个话题是什么”，还要判断“这个话题适合怎么讲、适合谁讲、适合讲给谁听”。

例如同样是高考分数线话题，标签系统不只输出“高考”，还会进一步识别：

- 领域：教育升学
- 具体子类：高考分数线
- 受众：学生、家长
- 内容形式：数据整理、志愿填报攻略、实时跟进
- 风险提示：教育信息需要核对官方来源
- 后续跟进：志愿填报时间、录取批次变化、考生反馈

从技术包装上看，这一模块后续可以演进为一个“Topic Semantic Tagging Engine”。它可以由以下能力组成：

| 能力 | 当前实现 | 后续可升级方向 |
| --- | --- | --- |
| 受控标签体系 | 规则词典和固定 taxonomy | 标签本体、标签图谱、人工审核后台 |
| 语义归类 | 标题和详情文本规则匹配 | 向量召回、Embedding 相似度、LLM 分类审核 |
| 实体识别 | 规则抽取关键词 | NER 实体识别、品牌/人物/地点标准化 |
| 事件识别 | 事件关键词匹配 | 事件 schema、时间线抽取、事件状态追踪 |
| 创作者匹配 | 内容模式和受众规则 | 创作者画像匹配、选题推荐排序 |
| 风险控制 | 风险词和领域规则 | 高风险话题审核、权威来源校验、多源一致性评分 |

这套标签体系的价值在于：下游项目拿到的不再是一个热榜标题，而是一个带语义、带证据、带创作建议、带风险提示的话题对象。

最终输出的 `creator_topic_index.json` 可以作为创作者项目的标准话题索引。它既能用于检索，也能用于推荐，还可以作为后续 Agent 自动生成选题建议时的结构化上下文。

## 6. 当前已经实现的功能

目前项目已经完成了从热榜发现到标签报告的第一阶段闭环。

已实现能力包括：

- 多平台热榜采集
- 今日和近 7 天窗口采集
- 核心平台配置
- 百度、微博、小红书、Bilibili、掘金详情策略
- 小红书外部热榜补充
- 话题基础去重与聚类
- 详情证据结构化保存
- JSON/JSONL 原始数据输出
- Markdown 热点报告输出
- 创作者热点语义索引
- 受控标签和动态关键词分层
- 话题风险等级与证据可追溯标记
- 本地缓存机制

## 7. 当前难点

### 7.1 热榜标题不等于话题详情

大部分热榜 API 只能提供标题、排名和热度，无法直接提供详细背景。对于创作者标签来说，这远远不够。

例如，小红书榜单可能只显示一个话题标题，但真正有价值的信息在相关笔记里，包括：

- 用户为什么讨论它
- 主要观点是什么
- 具体场景是什么
- 是否有争议
- 是否有品牌、产品、人物、地点等实体
- 适合哪类创作者跟进

这些信息必须二次采集和总结。

### 7.2 小红书等平台详情采集难度高

小红书、微博等平台存在登录态、动态渲染、风控、验证码等问题。当前项目不会绕过平台规则，而是采用登录态复用、低频访问、缓存优先和失败记录的方式。

这意味着短期内小红书正文详情不会完全依赖当前项目直接抓取，而是先保留话题和占位，后续由专门的小红书笔记采集项目补齐。

### 7.3 检索词划分仍需优化

当前已经实现基础查询词：

- 标题
- 标题 + 怎么回事
- 标题 + 最新进展

但复杂热点还需要更精细的拆分。例如：

- 人名、地名、品牌、产品名
- 事件动作词
- 时间信息
- 平台话题词
- 同义表达
- 多平台标题别名

这部分后续需要结合规则、模型或搜索反馈进一步优化。

### 7.4 真实性和准确性仍需增强

当前项目已经具备证据保存和来源回溯能力，但还没有完整实现多源事实核验。

后续需要补充：

- 权威来源识别
- 多平台交叉验证
- 新闻源优先级
- 时间线一致性检查
- 低可信内容过滤
- AI 摘要结果与原文证据对齐

## 8. 为什么采用当前技术路线

当前技术路线不是直接使用一个通用 Agent 完成所有事情，而是先建设稳定的数据管道。

原因如下：

1. 热点采集是持续任务，需要每天或每周稳定运行。
2. 下游项目需要结构化标签，而不是一次性自然语言回答。
3. 创作者标签需要可复用、可筛选、可追溯。
4. 单纯搜索 Agent 容易给出摘要，但不一定保存完整证据链。
5. 中文社交平台数据具有强平台差异，需要分别处理。

因此，当前采用：

- DailyHotApi 做第一层热点发现。
- 平台详情采集器做第二层内容补充。
- DetailEvidence 做第三层证据存储。
- 规则分类器做第四层标签生成。
- 后续再封装成 Skill 或 Agent 做自动化调度。

## 9. 后续 Skill / Agent 化方向

后续可以把当前项目封装成一个专用 Skill 或 Agent，形成自动化工作流。

建议的能力包括：

### 9.1 自动采集任务

支持：

- 每日热点采集
- 每周热点汇总
- 指定平台采集
- 指定垂类采集
- 强制刷新缓存
- 增量采集

### 9.2 自动详情补全

对每个热点自动判断缺失信息：

- 是否缺百度详情
- 是否缺微博讨论
- 是否缺小红书笔记
- 是否只有标题没有正文
- 是否需要搜索更多来源

### 9.3 自动语义标注

后续 Agent 不只是“自动打标签”，而是可以内置一个语义标注工作流：

```text
证据读取
→ 话题语义解析
→ 受控标签映射
→ 动态关键词抽取
→ 创作者意图匹配
→ 质量评分与复核提示
```

自动语义标注会输出：

- 垂类路径：话题属于哪个领域、哪个子类、哪个具体场景。
- 内容模式：适合做数据整理、攻略、科普、观点、复盘还是实时跟进。
- 受众标签：话题主要影响学生、家长、打工人、技术从业者、投资者还是泛大众。
- 实体关键词：人物、地点、品牌、产品、机构、考试、政策等。
- 事件关键词：发生了什么，例如发布、通报、处罚、上市、涨价、争议等。
- 检索短语：供下游系统召回相似话题或相关素材。
- 风险等级：是否涉及公共安全、教育升学、法律案件、医疗健康、金融投资等高敏领域。
- 创作者适配分：这个话题是否适合创作者跟进，以及适合哪类创作者跟进。

为了保证标签质量，Agent 后续还可以增加两类机制：

- 标签治理机制：所有核心标签必须落在受控 taxonomy 内，避免模型自由生成一堆不可维护的标签。
- 语义审核机制：当规则置信度低、证据不足或话题高风险时，自动标记为“待人工确认”。

### 9.4 自动质量检查

对生成结果做检查：

- 标签是否过粗
- 详情是否为空
- 是否只有热榜标题
- 是否缺少证据链接
- 是否存在高风险话题但没有权威来源
- 是否需要人工复核

### 9.5 输出给下游项目

最终可以输出稳定的数据文件或接口，例如：

```text
data/processed/creator_topic_index.json
data/evidence/detail_evidence_raw.jsonl
reports/creator_topic_cards.md
```

下游创作者项目可以直接消费 `creator_topic_index.json`，把其中的话题标签作为创作内容的标签参考。

## 10. 阶段性结论

当前项目已经完成了第一阶段关键验证：热点话题可以从多个平台持续采集，详情证据可以结构化保存，话题可以被初步分类并生成创作者标签。

目前最重要的认知是：

> 热榜 API 只能解决“发现热点”，不能解决“理解热点”。要为创作者提供正确标签，必须补充详情采集、证据回溯和标签分类。

因此，当前方案选择 DailyHot 作为热点发现入口是合理的，但它只是第一层。真正的项目价值在于后续的详情补全、真实性校验和创作者标签生成。

下一阶段建议重点推进：

1. 优化检索词拆分，提高详情召回准确度。
2. 补齐小红书笔记详情采集链路。
3. 引入新闻源和搜索 API 做多源交叉验证。
4. 完善标签体系，使其更适合下游创作者项目。
5. 将当前流程封装为可定时运行的 Skill 或 Agent。
